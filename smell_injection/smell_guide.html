Injected smells — reference guide

The 12 code smells in the benchmark, how each is injected, and real clean→smelly examples from the dataset. Green = added by the injector, red = removed.

Jump to:
mutable_default broad_except dead_code long_parameter_list deep_nesting complex_conditional long_method magic_number perf_try_in_loop inefficient_loop inefficient_copy duplicate_code
How to read this. Every smell is defined operationally: it is whatever its detector flags (pylint, ruff, or jscpd). Every injection is behaviour-preserving — the clean and smelly halves pass exactly the same tests — which is what makes each a valid labelled pair. The three groups below differ only in how natural the result looks, which is itself a consequence of those two rules.

De-optimisers

These take a real, well-written code shape and rewrite it as a worse one. They look the most like genuine bad code because they ARE genuine bad code -- just applied deliberately.

perf_try_in_loopruff

confirmed by: PERF203 (try-except-in-loop)
What it is
Putting a try/except inside a loop adds per-iteration overhead; the handler is usually better hoisted outside the loop.
How it is injected
The injector finds an existing loop and wraps its body in try: <body> except ValueError: pass. It uses ValueError (not Exception) on purpose, so it does not ALSO count as a broad_except.
Behaviour preserved?
Yes. If the loop body raises no ValueError, the try is transparent. (Applies only where a loop already exists.)
clean
def fifth_Power_Sum(n):


    sm = 0


    for i in range(1, n + 1):


        sm = sm + i * i * i * i * i


    return sm

inefficient_loopruff

confirmed by: PERF401 (manual-list-comprehension)
What it is
An inefficient loop: building a list with a manual append loop where a list comprehension would be clearer and faster.
How it is injected
The injector finds a transforming comprehension xs = [f(t) for t in it] and rewrites it as xs = [] then for t in it: xs.append(f(t)).
Behaviour preserved?
Yes. It builds the identical list, just the slow way.
clean
def sort_sublists(input_list):


    result = [sorted(x, key=lambda x: x[0]) for x in input_list]


    return result

inefficient_copyruff

confirmed by: PERF402 (manual-list-copy)
What it is
Copying a list with a manual loop where list(it) or it.copy() would do.
How it is injected
The injector finds a list copy ([x for x in it], list(it), it[:], or it.copy()) and rewrites it as an empty-list-plus-append loop. A guard (_provably_list) only allows this when it is provably a list, so a dict/set copy is never mis-rewritten.
Behaviour preserved?
Yes -- and the provably-a-list guard is exactly the fix for an earlier bug where a dict copy got turned into a broken list loop.

Extenders

These bolt a realistic-looking bad construct onto working code (an extra parameter, a dead import, a copy-pasted function). Natural enough to pass as something a tired programmer might actually write.

mutable_defaultpylint

confirmed by: W0102 (dangerous-default-value)
What it is
A function parameter whose default value is a mutable object, e.g. def f(x, cache=[]). Python builds that list ONCE and shares it across every call, so it silently accumulates state between calls -- a classic source of bugs.
How it is injected
The injector appends a new parameter cache=[] to the function's signature.
Behaviour preserved?
Yes. The added parameter is never referenced in the body, so the function computes exactly what it did before.
clean
def closest_num(N):


    return N - 1

smelly — mutable_default

    return N - 1

broad_exceptpylint

confirmed by: W0718 / W0703 (broad-except)
What it is
Catching bare Exception swallows every error -- including bugs you never meant to hide -- which makes failures silent and debugging painful.
How it is injected
The injector wraps the whole body (keeping any docstring outside) in try: <body> except Exception: return None.
Behaviour preserved?
Yes, for our tested code. If the original body raised nothing, the try runs identically. If it WOULD have raised, the clean twin fails its test too -- so the pair stays matched.

def closest_num(N):


    return N - 1
mbpp_89
smelly — broad_except

        return N - 1


    except Exception:


        return None

dead_codepylint

confirmed by: W0611 (unused-import) or W0612 (unused-variable)
What it is
Code that is present but has no effect: an unused import or an unused local variable. It clutters the file and misleads readers about what the function actually needs.
How it is injected
The injector adds one of two forms (it alternates, so the dataset contains both): an unused standard-library import (import os), or an unused local variable assigned from the first parameter (leftover = nums).
Behaviour preserved?
Yes. An unused import has no effect on the output, and the unused variable is assigned but never read.

duplicate_codejscpd

confirmed by: clone ≥ 5 lines / 25 tokens
What it is
Copy-pasted code. Two near-identical blocks mean every future fix has to be made in two places -- and one will be forgotten.
How it is injected
The injector appends a renamed copy of the whole function (foo plus foo_copy), creating a duplicated block that jscpd detects.
Behaviour preserved?
Yes. The copy is a separate, never-called function, so the original behaviour is untouched.
clean
def sort_mixed_list(mixed_list):


    int_part = sorted([i for i in mixed_list if type(i) is int])


    str_part = sorted([i for i in mixed_list if type(i) is str])


    return int_part + str_part

Minimal trip-wires

These add the SMALLEST inert construct that trips the detector's rule, while guaranteeing the code still behaves identically. They look synthetic on purpose -- a 'real' version of these smells would be load-bearing, and changing behaviour would break the clean/smelly pair.

deep_nestingpylint

confirmed by: R1702 (too-many-nested-blocks, >5)
What it is
Code nested many levels deep (loops inside loops inside ifs...) is hard to follow and hard to test.
How it is injected
The injector wraps the body in six nested for _n in range(1) loops. Each loop runs exactly once, so the real code still executes a single time -- just buried six levels deep.
Behaviour preserved?
Yes. range(1) means every loop iterates once, so the body runs exactly as before.
clean
def closest_num(N):


    return N - 1
mbpp_89
smelly — deep_nesting

    for _n5 in range(1):


        for _n4 in range(1):

                for _n2 in range(1):


                    for _n1 in range(1):


                        for _n0 in range(1):


                            return N - 1

complex_conditionalpylint

confirmed by: R0916 (too-many-boolean-expressions, >5)
What it is
A single condition chaining many boolean terms (a tangled compound conditional) is hard to read and easy to get subtly wrong.
How it is injected
The injector inserts if p and p and p ... (8 times): pass using the first parameter p. Eight terms clears pylint's limit of five.

Yes. The if body is just pass and the condition has no side effects, so nothing changes.

magic_numberpylint

confirmed by: R2004 (magic-value-comparison)
What it is
An unexplained literal number in the logic -- the reader has no idea what it means or why it is that value.
How it is injected
The injector inserts _ = p == 42 (comparing the first parameter against the bare literal 42).
Behaviour preserved?
Yes. The comparison's result is discarded into _.
Worth knowing: This is the one that looks most like dead code, and that is unavoidable. pylint's rule R2004 only fires on a magic number used in a COMPARISON (it would ignore price * 1.08), so the injector must produce a comparison. And a REAL magic number is load-bearing -- it sits inside the logic and drives the output -- so making it real would change behaviour and break the clean/smelly pair. The only way to be both detector-confirmed AND behaviour-preserving is an inert comparison, which necessarily reads as dead code. It is correctly labelled; it just looks synthetic by design.
clean
def closest_num(N):


    return N - 1
mbpp_89
smelly — magic_number
def closest_num(N):


    _ = N == 42


    return N - 1
